# 深度调研报告：OpenAI Codex / Hermes Agent / InkOS — Harness工程与Agent工程

## 调研目标

深入理解三个项目的 **Harness工程**（任务编排、流水线、验证、状态管理）和 **Agent工程**（Agent定义、生命周期、工具系统、配置、安全沙箱），为Thalia/Mnemosyne项目提供架构参考。

## 二、Harness工程（任务编排与流水线）

### 2.1 OpenAI Codex — Turn-Based Execution Pipeline

**核心模型**：基于Turn的执行循环

```
User Input → Turn Start → Context Assembly → Model Inference → 
  Tool Calls → Execution → Result Processing → Turn Complete
```

**关键机制**：

1. **Context Manager** — 汇集所有模型可见上下文，按fragment类型分slot，每个fragment有token预算上限（<10K tokens）
2. **Auto-Compaction** — 上下文超出token预算时自动压缩历史（`auto_compact_token_limit`触发）
3. **Tool Router** — 路由工具调用到对应runtime，支持并行执行
4. **Rollout Recording** — 每次交互持久化为JSONL，支持从任意checkpoint恢复和fork

**多Agent编排**：
- `ThreadManager` 管理多线程，支持fork（从历史状态创建新线程）
- 父子线程关系追踪，深度限制（`agents.max_depth`）
- `MultiAgentVersion` V1/V2支持协议演进

**错误恢复**：
- 沙箱拒绝 → 请求升级审批 → 无沙箱重试（如果策略允许）
- Turn中断 → `Op::TurnInterrupt` → 注入`TurnAborted`上下文

### 2.2 Hermes Agent — Tool-Call Iteration Loop

**核心模型**：同步while循环 + 迭代预算

```python
while (api_call_count < max_iterations and budget.remaining > 0):
    response = client.chat.completions.create(model, messages, tools)
    if response.tool_calls:
        execute_tool_calls(agent, response, messages)
        api_call_count += 1
    else:
        return response.content
```

**关键机制**：

1. **Iteration Budget** — 可配置的工具调用轮次上限，带一轮grace
2. **Concurrent Execution** — ThreadPoolExecutor（max 8 workers）并行执行工具调用
3. **Context Compression** — 可插拔`ContextEngine` ABC，支持第三方压缩引擎
4. **Steer Injection** — 用户可在轮次中途发送`/steer`注入新指令

**多Agent编排**：
- `delegate_task` 工具生成隔离子Agent，独立session_id、工具集、迭代预算
- Kanban Board — `kanban_tools.py`提供板式任务协调，工人通过环境变量获取任务

**错误恢复**：
- 空响应重试（3次后fallback到备选provider）
- API错误分类 → `FailoverReason` → 激活fallback链
- Thinking-only prefill（推理模型无可见输出时的恢复）

### 2.3 InkOS — Multi-Stage Chapter Pipeline

**核心模型**：7阶段结构化流水线（每章）

```
Planner → Composer → Writer → LengthNormalizer → ContinuityAuditor → [Reviser] → Persist
```

**关键机制**：

1. **Input Governance** — `ContextPackage` + `RuleStack` + `ChapterTrace` 精确追踪LLM看到了什么、为什么选择、token预算多少
2. **37维质量审计** — ContinuityAuditor跨37个维度评分（OOC、时间线、世界观、力量体系、钩子、节奏、风格...）
3. **State Settlement** — 写完后专用"settler"Agent解析章节内容，生成结构化`RuntimeStateDelta`（钩子操作、状态补丁、章节摘要）
4. **Scheduler** — 自动化守护进程模式，支持cron调度、质量门追踪、连续失败暂停

**多Agent编排**：
- 10+专业Agent（Architect, Planner, Composer, Writer, Auditor, Reviser, Polisher, LengthNormalizer, Radar, StateValidator）
- 基于`BaseAgent`抽象类，每个Agent有独立的系统提示和输出解析器
- 交互式Agent（pi-agent-core）与流水线Agent分离

**错误恢复**：
- Planner memo重试（3次，错误反馈注入提示）
- Architect不完整基础 → `ArchitectIncompleteFoundationError` 携带部分内容+缺失章节用于重试
- 审计→修订循环（可配置最大重试次数，每次重试温度递增0.1）
- 连续失败追踪 → Scheduler暂停书籍

## 三、Agent工程（Agent定义、工具、配置、安全）

### 3.1 OpenAI Codex

**Agent定义**：
- `Codex` → `Session` → `CodexThread` 三层抽象
- Agent状态机：`PendingInit → Active → Idle → [Active | Completed | Failed]`
- 每个Thread有独立的conversation history、token usage、approval state

**工具系统**：
```
ToolSpec (定义) → ToolExecutor (trait) → ToolRouter (路由) 
  → ToolOrchestrator (审批+沙箱) → ToolRuntime (执行)
```
- 内置工具：shell, apply_patch, web_search, request_user_input, tool_search, request_plugin_install
- MCP工具：动态发现，OAuth支持
- 工具发现：`tool_search` 元工具让模型查询可用工具

**配置系统**：
- 7层配置合并：默认值 → 用户配置 → Profile → 项目配置 → 云配置 → CLI覆盖 → 运行时覆盖
- `ConfigToml` 100+字段，包含模型、安全、Agent、工具、上下文、扩展、基础设施
- Named Profiles支持（`[profiles.coding]`, `[profiles.research]`）

**安全沙箱**：
| 平台 | 机制 |
|------|------|
| macOS | Seatbelt (sandbox-exec) |
| Linux | Bubblewrap + Landlock/seccomp |
| Windows | Restricted Token + AppContainer |
- 网络代理：MITM强制域名级策略
- Guardian：自动代码审查Agent，可替代人工审批
- ExecPolicy：Starlark策略引擎，命令级allow/deny

### 3.2 Hermes Agent

**Agent定义**：
- `AIAgent` 类，~60个构造参数
- 两个入口：`chat(message)` 简单接口 / `run_conversation()` 完整接口
- 状态：`_cached_system_prompt`, `_memory_store`, `_todo_store`, `iteration_budget`

**工具系统**：
- **自动发现注册**：`tools/*.py`文件导入时自动`registry.register()`
- **Toolset分组**：`web`, `terminal`, `file`, `debugging`, `hermes-cli` 等可组合包
- **Footprint Ladder**：6级能力添加决策框架（从"扩展现有代码"到"新核心工具"）
- 40+核心工具：web_search, terminal, read_file, write_file, patch, delegate_task, kanban_*, memory, todo

**配置系统**：
- `~/.hermes/config.yaml` + `~/.hermes/.env`
- ProviderProfile数据类：声明式provider描述，含hooks（prepare_messages, build_extra_body, fetch_models）
- 三层系统提示：STABLE（身份，字节稳定用于缓存）→ CONTEXT（AGENTS.md）→ VOLATILE（内存快照）

**安全模型**：
- 6种终端后端：Local, Docker, SSH, Singularity, Modal, Daytona
- 审批系统：危险命令需用户确认
- Guardrails：per-tool策略评估
- Path Security：路径穿越防护
- 文件安全：跨profile写保护

### 3.3 InkOS

**Agent定义**：
- **双层Agent架构**：
  - Layer 1: 流水线Agent（BaseAgent子类，无状态，每次调用）
  - Layer 2: 交互式Agent（pi-agent-core Agent，有状态对话）
- 10+专业Agent，每个有独立系统提示和输出解析器

**工具系统**：
- TypeBox JSON Schema定义
- **propose_action门控**：生产操作需用户确认
- SessionKind动态工具集：chat/book/short/play/edit模式各有不同工具可用
- `sub_agent`工具委托给流水线Agent

**配置系统**：
- 4层配置：全局config.json → 项目inkos.yaml → 书籍book.json → 类型profile YAML
- GenreProfile：类型特定规则（疲劳词、节奏、审计维度）
- 模型覆盖：per-agent模型路由（`modelOverrides`）
- 程序化构建系统提示（非模板），支持双语（zh/en）

**安全模型**：
- `propose_action`工具门控生产操作
- `isConfirmed()`检查操作来源（按钮/斜杠）是否匹配请求意图
- `architectCreateOnly`标志限制书籍创建会话只能调用architect
- 无OS级沙箱

## 四、核心设计模式对比

### 4.1 状态管理

| 模式 | Codex | Hermes | InkOS |
|------|-------|--------|-------|
| **持久化** | SQLite 4库 + JSONL rollout | SQLite WAL + FTS5 | Markdown文件 + SQLite + Zod结构化JSON |
| **恢复** | JSONL重放任意checkpoint | SessionDB自动恢复 | 文件锁 + PID检查 |
| **压缩** | Auto-compact（token预算触发） | 可插拔ContextEngine | Truth文件>6000字符压缩为标题索引 |
| **追踪** | W3C trace context | ContextVar传播到worker线程 | ChapterTrace记录上下文选择原因 |

### 4.2 工具系统

| 模式 | Codex | Hermes | InkOS |
|------|-------|--------|-------|
| **定义** | Rust trait + JSON Schema | Python registry.register() | TypeBox JSON Schema |
| **发现** | tool_search元工具 | 自动导入发现 | SessionKind过滤 |
| **审批** | ToolOrchestrator + Guardian | approval.py + guardrails | propose_action + isConfirmed() |
| **并行** | ToolCallRuntime | ThreadPoolExecutor(8) | 无（顺序执行） |
| **扩展** | MCP + 插件 + skills | toolset分组 + plugins | 无 |

### 4.3 提示管理

| 模式 | Codex | Hermes | InkOS |
|------|-------|--------|-------|
| **构建** | ContextFragment slot系统 | 三层（STABLE/CONTEXT/VOLATILE） | 程序化构建，6种SessionKind |
| **缓存** | Fragment级token预算 | 字节稳定不变（节省75%输入token） | Truth文件每次turn从磁盘读取 |
| **注入** | PromptSlot位置注入 | 用户消息注入（非系统提示） | ContextTransform注入truth文件 |
| **多语言** | 英文 | 英文 | 双语（zh/en） |

## 五、对Thalia/Mnemosyne的启示

### 5.1 Harness工程启示

1. **InkOS的7阶段流水线最接近Thalia的需求**：Plan → Compose → Write → Audit → Revise → Reflect，Thalia的8-agent pipeline可以直接参考
2. **Input Governance模式**（ContextPackage + RuleStack + ChapterTrace）提供了完整的审计线索，知道LLM看到了什么、为什么
3. **State Settlement**（写完后settler Agent提取结构化状态变更）解决了"叙事与状态管理分离"的问题
4. **质量门+连续失败暂停**是生产级流水线的必要机制
5. **温度递增重试**（每次重试+0.1）是简单有效的探索策略

### 5.2 Agent工程启示

1. **双层Agent架构**（流水线Agent + 交互式Agent）值得采用：流水线Agent无状态可测试，交互式Agent有状态可对话
2. **propose_action门控**是平衡自动化与用户控制的好模式
3. **SessionKind动态工具集**比固定工具集更安全
4. **Footprint Ladder**（6级能力添加框架）防止工具膨胀
5. **可插拔ContextEngine/MemoryProvider**提供扩展性

### 5.3 安全启示

1. **Codex的OS原生沙箱**是最高安全级别，但实现复杂
2. **Hermes的6种终端后端**提供了灵活的隔离选择
3. **InkOS的propose_action门控**是最轻量的安全模型，适合本地应用
4. **Guardian自动审查**（Codex）是减少人工审批负担的好方案

### 5.4 架构参考优先级

| 优先级 | 参考来源 | 内容 |
|--------|---------|------|
| **P0** | InkOS | 7阶段流水线、Input Governance、State Settlement、双层Agent |
| **P1** | Hermes | Tool自动发现注册、Footprint Ladder、可插拔ContextEngine、三层提示 |
| **P2** | Codex | Extension系统（10+hook点）、JSONL rollout、多层配置、MCP集成 |
| **P3** | Codex | OS原生沙箱、Guardian自动审查（生产级安全时参考） |

## 六、下一步行动

基于以上调研，Thalia/Mnemosyne的harness工程和agent工程应：

1. **采用InkOS的流水线模式**：定义7-8个阶段的章节流水线，每个阶段一个专业Agent
2. **实现Input Governance**：ContextPackage + RuleStack + ChapterTrace，精确追踪上下文来源
3. **实现State Settlement**：写完后用专用Agent提取结构化状态变更（钩子、角色、世界观）
4. **采用双层Agent架构**：流水线Agent（BaseAgent子类）+ 交互式Agent（对话式）
5. **实现propose_action门控**：生产操作需用户确认
6. **参考Hermes的工具注册模式**：自动发现 + toolset分组
7. **参考Codex的Extension系统**：10+生命周期hook点，支持插件扩展
